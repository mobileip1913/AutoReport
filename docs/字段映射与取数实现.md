# 字段映射 → 输出不同数据 的实现原理

> 本文专讲：财务在 Web 上**配置字段映射**后，系统是如何据此从 Excel 数据里**算出不同指标值**的。
> 这是 AutoReport 的核心机制，也是「列头/口径变更不依赖研发」的技术实现。

---

## 1. 核心思想：把「怎么取数」做成数据，而不是代码

传统做法是把计算逻辑写死在 Excel 公式或脚本里——列头一变就得改代码。

AutoReport 把「取数逻辑」存成**数据库里的配置**（`field_mappings` + `field_mapping_parts`）。
计算引擎是**通用**的：它不认识「销售额」「应收金额」，只会读配置、按配置去 `data_rows` 里过滤、聚合、组合。

```
配置（数据）          通用引擎（代码，不变）         结果
─────────────        ──────────────────         ──────
读哪个文件/列    ┐
怎么过滤行       ├──►  按配置执行取数  ──────►   逻辑字段值
怎么聚合/组合    ┘                              {code: 数值}
```

**所以：改口径 = 改配置（数据），不需要改引擎（代码）。**

---

## 2. 三个概念的关系

```
逻辑字段 (logical_field)         例：mc_actual_payment「实际支付金额」
   │  跨平台统一的业务指标，与具体 Excel 无关
   │
   └─ 字段映射 (field_mapping)   「这个数据源下，该逻辑字段怎么取数」
        │
        └─ 取数规则 (field_mapping_part) ×N   一条或多条，可加减组合
             每条 part 回答 4 个问题：
               ① 从哪取？ 文件关键字 + Sheet + 列头
               ② 取哪些行？ 日期过滤 + 行条件 + 样品/刷单排除 + 关联订单
               ③ 怎么汇总？ sum / count_distinct / sum_dedup ...
               ④ 和别的 part 怎么合？ 加 / 减
```

一个逻辑字段可由**多条 part 加减组合**。例：

> **应收金额** = `SKU Subtotal After Discount`(求和) **＋** `SKU Platform Discount`(求和)
> 就是 2 条 part，`combine_op` 分别是 add、add。

---

## 3. 一条 part 的完整字段（即「可配置项」）

对应 `FieldMappingPart` 模型，也就是 Web 弹窗里的每一项：

| 配置项 | 字段 | 作用 |
|--------|------|------|
| 来源文件关键字 | `source_file_keyword` | 按文件名筛选（订单/结算/退货…），空=全部文件 |
| Sheet | `sheet_name` | 取哪个工作表 |
| 列头 | `column_header` | 取哪一列的值 |
| 列头别名 | `aliases` | 防列头漂移，任一命中即可 |
| 聚合方式 | `aggregation` | sum / count / count_distinct / sum_dedup / max_dedup / avg |
| 去重键 | `dedup_keys` | 去重/分组依据，如 `Order ID` |
| 组合运算 | `combine_op` | 与上一条 part 的关系：add / subtract |
| 按日期列过滤 | `date_filter_column` + `date_format` | 只保留「该列日期 = 报表日期」的行 |
| 行过滤条件 | `row_filters` | eq/ne/in/not_in/gt/gte/lt/lte/nonempty |
| 排除样品单 | `exclude_sample` | 剔除「整单 SKU 总额=0」的订单 |
| 排除刷单单 | `exclude_review` | 剔除外部刷单清单里的订单 |
| 关联当日有效订单 | `join_to_orders` | 只保留命中当日有效订单的行（用于佣金等关联表） |

## 4. 配置是怎么存进去的（Web → DB）

### 4.1 前端收集（`app/static/mapping_modal.js`）

弹窗里每条规则收集成一个对象，保存时 POST/PUT 给后端：

```javascript
// 每条 part 的数据
{
  sheet_name: "OrderSKUList",
  column_header: "Order Amount",
  aggregation: "sum_dedup",
  dedup_keys: ["Order ID"],
  combine_op: "add",
  date_filter_column: "Created Time",
  date_format: "us",
  exclude_sample: true,
  exclude_review: true,
  ...
}
```

### 4.2 后端落库（`app/routers/api.py`）

`PUT /api/mappings/{id}` 或 `POST /api/mappings`：先删掉该映射旧的 parts，再按提交内容重建。

```python
def _apply_parts(mapping, parts, db):
    db.query(FieldMappingPart).filter(FieldMappingPart.mapping_id == mapping.id).delete()
    for idx, part in enumerate(parts):
        db.add(FieldMappingPart(mapping_id=mapping.id, sort_order=idx, ...))
```

> 配置存的是「**规则**」，不是「结果」。改完立即对下一次出报生效，已生成的历史报表不变。

## 5. 引擎是怎么按配置算出值的（DB → 数值）

入口：`report_engine.aggregate_field_values()`。对每个逻辑字段执行三步——**过滤 → 聚合 → 组合**。

### 5.1 取出数据池

日报模式下，跨该店铺**所有导入**取出全部 `data_rows`（一个月的数据池），再靠 part 上的「日期过滤」筛出当日。

```python
rows = db.query(DataRow).filter(DataRow.data_import_id.in_(import_ids)).all()
context = build_daily_context(rows, data_source.config, report_date)  # 算样品集/刷单集/当日有效订单
```

### 5.2 逐条 part：过滤行（`_filter_rows`）

按配置层层筛选每一行：

```
该行 sheet == part.sheet_name ?
   ↓ 是
文件名包含 source_file_keyword ?
   ↓ 是
该行 date_filter_column 的日期 == 报表日期 ?   （多格式：us/eu/iso）
   ↓ 是
不在样品订单集 / 不在刷单订单集 ?
   ↓ 是
（如需）命中当日有效订单键 ?
   ↓ 是
满足所有 row_filters ?
   ↓ 是
✓ 这一行参与聚合
```

### 5.3 逐条 part：聚合（`aggregate_part`）

按 `aggregation` 把筛出的行汇总成一个数：

| 聚合方式 | 行为 | 典型用途 |
|----------|------|----------|
| `sum` | 所有行的列值相加 | SKU 级折扣 |
| `count` | 行数 | — |
| `count_distinct` | 按去重键统计唯一值个数 | **订单数**（同单多 SKU 算 1） |
| `sum_dedup` | 按去重键分组，每组只取一次再相加 | **实际支付**（整单金额在每行重复，去重避免重复累加） |
| `max_dedup` | 每组取最大再相加 | — |
| `avg` | 平均值 | — |

```python
if agg == "sum_dedup":          # 例：Order Amount 按 Order ID 去重求和
    groups = {}
    for row in matched:
        key = _dedup_key(row.row_data, dedup_keys)   # 如 (Order ID,)
        groups[key] = _cell(row.row_data, column_header, aliases)
    return sum(groups.values())
```

### 5.4 part 间组合（`combine_parts`）

一个字段的多条 part 算完后，按 `combine_op` 加减：

```python
# 应收金额 = part0(add) + part1(add)
# 净额示例   = part0(add) - part1(subtract) - part2(subtract)
total = 0
for part, value in zip(parts, part_values):
    total += value if part.combine_op != "subtract" else -value
```

最终得到 `field_values = {"mc_actual_payment": 6475.35, "mc_receivable_amount": 6429.18, ...}`。

## 6. 从字段值到报表（套模板公式）

报表模板的每个指标行有一个**表达式**，引用逻辑字段：

```
实际支付金额   {field:mc_actual_payment}
应支付金额     ={field:mc_actual_payment}+{field:mc_payment_platform_discount}+{field:mc_sku_platform_discount}
下单数         ={field:mc_actual_order_count}-{field:mc_cancelled_order_count}-{field:mc_refunded_order_count}
```

`formula.py` 把 `{field:xxx}` 换成上一步算好的数值，再用 Python `ast` **安全求值**（白名单四则运算，不用 `eval`），最后按格式（usd/percent/integer）输出。

```
{field:...} 表达式  ──代入 field_values──►  纯算术式  ──AST 求值──►  数值  ──格式化──►  "$6,990.57"
```

## 7. 端到端示例：「实际支付金额」是怎么算出来的

**配置**（1 条 part）：

| 项 | 值 |
|----|----|
| 文件关键字 | `订单` |
| Sheet | `OrderSKUList` |
| 列头 | `Order Amount` |
| 聚合 | `sum_dedup` |
| 去重键 | `Order ID` |
| 日期过滤 | `Created Time`（us 格式）= 报表日期 |
| 排除样品 / 排除刷单 | ✅ / ✅ |

**执行**：

```
订单表 17630 行 SKU 明细
  → 只留 Created Time = 2026-06-22 的行
  → 剔除样品订单（整单 SKU 总额=0）→ 当日 244 单中去掉 4 单样品
  → 按 Order ID 去重（同单多 SKU 的 Order Amount 只取一次）
  → 求和
= $6,475.35   ← 逻辑字段 mc_actual_payment 的值
```

**进模板**：指标行「实际支付金额」表达式 `{field:mc_actual_payment}` → 显示 `$6,475.35`。

> 若财务把列头改成 `Order Amount1` 并重新导入，**只改配置里的列头**，结果立即变成 `$6,485.35`，引擎代码一行都不用动。

## 8. 「输出不同数据」的几种触发方式

| 想要不同的输出 | 改什么 | 是否要重新导入 |
|----------------|--------|----------------|
| 换取数列（如 Order Amount → Order Amount1） | 改 part 的列头 | 列已在库则否；新列需重导 |
| 换口径（去重/不去重、加 or 减） | 改 aggregation / combine_op | 否 |
| 加筛选（只算某状态/某仓库） | 加 row_filters | 否 |
| 排除样品/刷单 | 勾选 exclude_sample / exclude_review | 否 |
| 跨文件组合（订单 − 结算扣费） | 加一条 part + subtract | 否 |
| 换报表日期 | 生成时选日期 | 否（数据池已在库） |
| Excel 内容变了 | — | **是，必须重新导入** |

## 10. 一句话总结

> **配置定义「怎么取数」，通用引擎照着算，公式把字段拼成报表。**
> 改口径只改配置（数据库里的 part），不动引擎代码——这就是财务能自助、研发能解放的根本原因。
