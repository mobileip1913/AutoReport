# 字段映射与报表模板合并方案

> 状态：已确认，待实施  
> 目标：财务在一个页面完成「取数 + 报表展示 + 公式」，改完即生效；为后续「每人只看自己的报表」留扩展位。

---

## 1. 决策摘要

| 项 | 结论 |
|----|------|
| 公式行 | **保留**，与取数行同在「报表配置」页，财务可一眼看到完整日报结构 |
| 逻辑字段 | **降级为内部 code**，不单独维护；见 §2 |
| 数据源 | **每个数据源一套报表行**（非全局模板复用） |
| 发布流程 | **取消** publish/draft，保存即生效 |
| 权限（后续） | 报表行 / 数据源级 `owner_id`，本阶段只留字段 |

---

## 2. 逻辑字段怎么处理（建议）

**现状**：`logical_fields` 是全局指标注册表（`mc_actual_payment`），映射和模板都引用它。

**合并后**：

- 财务只看到 **指标名称**（如「实际支付金额」）
- 系统内部自动生成 **`line_code`**（如 `mc_actual_payment` 或 `line_12`），仅用于：
  - 公式引用 `{field:mc_actual_payment}`
  - 行与行之间复用（已有字段复用）
- **`logical_fields` 表保留但不再单独维护**：
  - 迁移期：`field_mappings.logical_field_id` 仍可关联旧数据
  - 新行：`label` + `line_code` 写在 mapping 上，`logical_field_id` 可空
  - 长期：可删 `logical_fields` 表（非本阶段必须）

**行类型 `line_type`**：

| 类型 | parts | expression | 说明 |
|------|-------|------------|------|
| `fetch` | 必填 | 空或 `{field:自身code}` | 从 MySQL 取数 |
| `formula` | 空 | `=...` 或 `{field:a}+{field:b}` | 纯公式行 |

## 3. 目标数据模型

在 **`field_mappings`** 上扩展（少建新表，兼容迁移）：

```text
field_mappings（语义上 = 报表行）
├── data_source_id
├── line_type          fetch | formula
├── label              指标名称（财务可见）
├── line_code          内部引用码，数据源内唯一
├── report_group       分组：订单情况 / 佣金 / …
├── sort_order         报表顺序
├── expression         公式行表达式；取数行可空
├── format_type        usd | integer | percent
├── is_highlight       是否高亮
├── owner_id           nullable，预留权限
├── description
└── parts[]            现有 field_mapping_parts（仅 fetch 行）

废弃/只读（本阶段）：
├── report_templates
├── template_lines
└── logical_fields（仅读，不新增）
```

**约束**：

- `(data_source_id, line_code)` 唯一
- `(data_source_id, sort_order)` 可重复 group 内排序
- 公式行：`parts` 为空；取数行：至少一条 part 或 ref_field

## 4. 目标 UI

**单页：报表配置**（原「字段映射」，导航改名）

```
[数据源 ▼ 美宠 TK-US]

── 订单情况 ──────────────────────────────
 #  指标名称      类型    规则摘要 / 公式              操作
 1  实际支付金额  取数    订单.OrderSKUList…          [配置]
 2  应支付金额    公式    =实际支付+平台折扣+SKU折扣    [编辑公式]
 3  应收金额      取数    …                           [配置]

── 佣金 ───────────────────────────────────
12  达人佣金      公式    =达人+服务商                  [编辑公式]
…

[+ 新增取数行]  [+ 新增公式行]  [预览日报]
```

**弹窗**：

- 取数行 → 现有 `mapping_modal.js`（parts）
- 公式行 → 轻量弹窗：表达式编辑器 + 可选引用本数据源已配行（下拉插入 `{field:code}`）

**移除/隐藏**：

- 导航「报表模板」及 `/templates/*`（或 302 到报表配置）
- 模板 publish / test 按钮

**日报输出 `/daily`**：

- 按选定 `data_source_id` 读该源下全部 `field_mappings` 排序出报
- 不再查 `MEICHONG_TEMPLATE_NAME`

## 5. 引擎改动要点

```
generate_report(data_source_id, date, store)
  1. 加载 field_mappings WHERE data_source_id ORDER BY sort_order
  2. fetch 行 → aggregate_field_values（按 line_code 存值）
  3. formula 行 → evaluate_expression(expression, values_by_code)
  4. 按 report_group 分组 → build_grouped（改读 mapping 而非 TEMPLATE_GROUPS）
  5. 写 report_runs / report_values（line_label = mapping.label）
```

`TEMPLATE_GROUPS` 静态配置 → 改为运行时按 `report_group` 聚合。

---

## 6. 迁移（美宠 ds_id=1）

1. 把 `TEMPLATE_LINES` + `MAPPINGS` 合并写入各 `field_mapping` 的 `label/group/sort/expression/format`
2. 取数行：`line_type=fetch`，parts 已有
3. 公式行：新建 mapping 行（无 parts），`line_type=formula`
4. 占位指标（刷单/广告等）：formula 或 fetch 空 parts，出报仍补 0

## 7. 后续权限（本阶段只预留）

## 9. 风险

| 风险 | 缓解 |
|------|------|
| 公式引用 code 变更导致公式失效 | code 创建后不可改，仅 label 可改 |
| 同一指标取数+公式拆两行 vs 一行 | 保持现有：取数用 fetch 行，展示用 formula 行可引用 |
| 迁移重复行 | 迁移脚本幂等，按 line_code upsert |
| mapping_modal.js 体积 | 公式编辑器独立小模块，不塞进 1000 行 parts 逻辑 |

---

## 10. 不在本阶段

- 用户登录与权限过滤
- 多租户 / 店铺级报表隔离
- 删除 `logical_fields` / `report_templates` 物理表（可先保留只读）
