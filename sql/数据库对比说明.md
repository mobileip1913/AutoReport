# 生产数据库 vs AutoReport 事实库对比说明

本文档对比 **生产环境 TikTok 跨境事实表**（见 `跨境订单建表语句_v2.sql`）与 **AutoReport 项目事实库**（`fact_mc_*` 及 Catalog 元数据）的差异，供 ETL 对接、字段对齐与后续迁移参考。

---

## 1. 总体架构

| 维度 | 生产库 | AutoReport |
|------|--------|------------|
| 定位 | 多店铺共用的事实数据仓库 | 报表自助出数 Demo / ETL 事实层 |
| 事实表数量 | 6 张固定表 | 8 张动态事实表 |
| 表名前缀 | `eb_overseas_tk_*` | `fact_mc_*` |
| 业务列来源 | TikTok Excel 导出，列名 snake_case | 同上，列名由 `slugify_header()` 动态生成 |
| 原始表头留存 | 字段 `COMMENT` 存 Excel 表头 | `catalog_columns.header_name` 存 Excel 表头 |
| 映射配置 | 无独立映射表，约定 + DDL 注释 | `catalog_files` / `catalog_sheets` / `catalog_columns` |

---

## 2. 事实表对照

生产库 6 段导出（A～E）与我们的 Sheet / 事实表对应关系如下。

| 段 | 业务含义 | 生产表名 | 我们的事实表 | Excel 文件 keyword | Sheet |
|----|----------|----------|--------------|-------------------|-------|
| A | 订单 | `eb_overseas_tk_order` | `fact_mc_order_sku` | 订单 | OrderSKUList |
| B | 退货退款单 | `eb_overseas_tk_order_return` | `fact_mc_return` | 退货退款单 | 0 |
| C1 | 联盟达人佣金 | `eb_overseas_tk_affiliate_creator` | `fact_mc_affiliate_creator` | 联盟达人佣金 | Sheet1 |
| C2 | 联盟服务商佣金 | `eb_overseas_tk_affiliate_partner` | `fact_mc_affiliate_partner` | 联盟服务商佣金 | Sheet1 |
| D | 结算表 | `eb_overseas_tk_finance_statements` | `fact_mc_settlement_details` | 结算表 | Order details |
| E | 未结算单 | `eb_overseas_tk_finance_on_hold` | `fact_mc_unsettled` | 未结算单 | Unsettled order and adjustment |

**我们多出的 2 张表（生产无对应）：**

| 事实表 | Sheet | 说明 |
|--------|-------|------|
| `fact_mc_settlement_statements` | Statements | 结算汇总 |
| `fact_mc_settlement_payments` | Payments | 付款记录 |

生产 SQL 仅对结算 Excel 的 **Order details** 建表；Statements / Payments 等 Sheet 未落库。

配置来源：`scripts/etl/meichong_dataset.py` 中的 `FILE_SHEETS`、`SHEET_SPECS`。

## 3. 公共扩展字段

两组字段均附加在每张事实表上，**不是** TikTok Excel 业务列。

### 3.1 生产库

| 字段 | 类型 | 用途 |
|------|------|------|
| `id` | BIGINT PK | 行主键 |
| `store_id` | INT UNSIGNED | 店铺 ID，外键关联店铺主表 |
| `excel_order_id` | INT UNSIGNED | 导出批次 ID，外键关联导出记录表 |
| `import_time` | INT | 数据导入时间（Unix 时间戳） |
| `shop_code` | VARCHAR(255) | 店铺业务编码，冗余便于查询 |
| `extra_data` | JSON | 动态字段池：各店铺不一致的扩展列 |

索引：`idx_store_id`、`idx_excel_order_id`、`idx_import_time`。

### 3.2 AutoReport

| 字段 | 类型 | 用途 |
|------|------|------|
| `id` | BIGINT PK | 行主键 |
| `data_source_id` | INT | 数据源 ID，关联 `data_sources`（含映射规则、Catalog） |
| `store_name` | VARCHAR(100) | 店铺名称，取数时 `WHERE store_name = ?` |
| `biz_date` | DATE | ETL 从业务列解析的业务日期（如 Created Time） |
| `etl_batch_id` | INT | ETL 批次 ID，关联 `etl_batches` |
| `created_at` | DATETIME | 行写入时间 |

定义见 `app/services/fact_storage.py` 的 `BASE_COLUMNS`；写入见 `scripts/import_meichong.py`。

### 3.3 概念对应

| 目的 | 生产 | AutoReport |
|------|------|------------|
| 标识店铺 | `store_id` + `shop_code` | `data_source_id` + `store_name` |
| 标识导入批次 | `excel_order_id` | `etl_batch_id` |
| 导入/写入时间 | `import_time` | `created_at` / `etl_batches.created_at` |
| 业务日期（日报筛选用） | 无公共列，用业务字段如 `created_time` | `biz_date`（ETL 预解析） |
| 店铺差异扩展列 | `extra_data` JSON | 无等价公共列 |

## 4. 业务字段与列名规则

### 4.1 对齐方式

- **生产**：DDL 中列名固定（如 `order_amount`），`COMMENT` 为原始 Excel 表头（如 `'Order Amount'`）。
- **我们**：Catalog 存 `header_name`（Excel 表头）→ `db_column`（物理列）；物理列可由 `app/services/column_naming.py` 的 `slugify_header()` 生成。

命名规则（与生产 SQL 注释一致）：

- 表头转小写
- 空格 → 下划线
- 特殊符号清理（如 `Phone #` → `phone_num` / 生产为 `phone_no`）

**注意**：完全对齐生产时，应以生产 DDL 列名为准，不能仅依赖 `slugify_header()` 推断。

### 4.2 各段业务列数量（与 `data/schemas/ds_4.json` 比对）

| 段 | 生产 COMMENT 列数 | 我们 Excel 列数 | 差异说明 |
|----|-------------------|-----------------|----------|
| A 订单 | 60 | 60 | 已对齐（曾有多余测试列 `Order Amount1`，已清理） |
| B 退货退款单 | 25 | 25 | 一致 |
| C1 联盟达人佣金 | 30 | 30 | 一致 |
| C2 联盟服务商佣金 | 30 | 30 | 一致 |
| D 结算 Order details | 103 | 102 | 生产多 `return_shipping_fee_paid_by_customers_2`（重复表头拆列） |
| E 未结算单 | 98 | 98 | 一致 |

## 5. 文件名 / Sheet / 字段映射

### 5.1 生产库

**没有**独立的「文件名 → Sheet → 字段」配置表。映射关系为：

- **文件 → 表**：按导出类型固定写入 6 张表之一（ETL 约定）
- **Sheet → 表**：基本 1 段导出对应 1 表；结算文件仅 **Order details** 落库
- **字段 → 列**：DDL 固定；`COMMENT` = 原始表头

批次与店铺通过 `excel_order_id`、`store_id`、`shop_code` 关联外部主数据（批次表、店铺表不在本 SQL 文件中）。

### 5.2 AutoReport

显式 Catalog 三层结构（`app/models.py`）：

```
catalog_files     keyword, file_name, file_label  →  逻辑「来源文件」
catalog_sheets    sheet_name, fact_table            →  Sheet → 事实表
catalog_columns   header_name, db_column, aliases   →  Excel 表头 → 物理列
```

字段映射 UI、取数引擎通过 `app/services/catalog_resolver.py` 解析上述配置。

Schema 快照可导出为 `data/schemas/ds_4.json`（`scripts/export_schema.py`）。

## 6. 重复列头处理（重要差异）

TikTok 导出 Excel 有时会出现**两列表头文字完全相同**的情况。

**示例（结算表 Order details）：**

| … | Return shipping fee (paid by customers) | … | Return shipping fee (paid by customers) | … |
|---|----------------------------------------|---|----------------------------------------|---|
| 行1 | 1.50 | … | 0.00 | … |

### 6.1 生产库：拆列

第二列单独建字段，例如：

- `return_shipping_fee_paid_by_customers`
- `return_shipping_fee_paid_by_customers_2`（COMMENT: `Return shipping fee (paid by customers) 2`）

两列数据均保留。

### 6.2 AutoReport：后列覆盖前列

`app/services/excel_reader.py` 将每行读入 `dict[header → value]`。同名列重复时，**后写入的值覆盖先写入的值**，最终只保留最后一列。

```python
for i, header in enumerate(headers):
    item[header] = row[i]  # 同 header 会覆盖
```

**影响**：若两列均有不同有效值，我们会丢失第一列，与生产数据可能不一致。

**订单表**曾出现两个 `Order Amount`；测试环境曾用 `Order Amount1` 表示第二列，已按生产标准清理，仅保留 `Order Amount`。

## 7. 数据类型与存储差异

| 项目 | 生产 | AutoReport |
|------|------|------------|
| 金额/费用/佣金 | `DECIMAL(18,4)` 等 | 事实表动态列多为 `TEXT`（MySQL）或 SQLite TEXT |
| 时间字段 | `DATETIME` | 业务列多为 TEXT；`biz_date` 为 DATE |
| 字符集 | utf8mb4_unicode_ci | utf8mb4（MySQL） |
| 店铺扩展字段 | `extra_data` JSON | 无；非常规列进 Catalog 映射的独立物理列 |

---

## 8. 差异汇总

| # | 差异项 | 生产 | AutoReport |
|---|--------|------|------------|
| 1 | 事实表数量 | 6 | 8（多 Statements、Payments） |
| 2 | 店铺标识 | 数字 `store_id` + `shop_code` | `data_source_id` + 字符串 `store_name` |
| 3 | 批次标识 | `excel_order_id` | `etl_batch_id` |
| 4 | 业务日期公共列 | 无 | `biz_date` |
| 5 | 映射元数据 | 无 Catalog 表 | `catalog_*` 三张表 |
| 6 | 重复列头 | 拆成 `_2` 等 | dict 覆盖，丢第一列 |
| 7 | 结算多 Sheet | 仅 Order details | Order details + Statements + Payments |
| 8 | 动态扩展列 | `extra_data` | 按 Catalog 扩列 |

## 9. 对齐建议（待决策）

按优先级：

1. **重复列头**：在 `excel_reader.py` 检测重复表头并自动后缀（如 ` 2`），与生产 `_*_2` 列对齐。
2. **列名白名单**：从生产 DDL 生成「表头 → db_column」静态映射，避免 `slugify_header` 与生产个别字段不一致（如 `phone_no` vs `phone_num`）。
3. **结算 Statements / Payments**：确认是否保留我们侧两张表，或仅跟生产保留 Order details。
4. **公共字段**：若直连生产库，需映射 `store_id` / `excel_order_id` ↔ 我们的 `data_source_id` / `etl_batch_id`；`biz_date` 为报表侧增值字段，生产无直接对应。
5. **表名**：对接时可继续用 `fact_mc_*` 做视图/同步层，或迁移至 `eb_overseas_tk_*`。
